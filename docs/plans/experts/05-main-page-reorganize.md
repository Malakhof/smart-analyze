Если первый экран не даёт триггера — РОП закрывает вкладку и идёт в CRM. Это первый и главный риск премьеры 4.05."

## Solution Options

### A: Frequency-based reorder (минимальный риск)

- **Суть:** просто переупорядочить блоки по частоте использования из anketa, без секций. Above-the-fold = Activity + Quality + Worst calls. Heatmap и Funnel — вниз.
- **Pros:** 2-3 часа работы, риск багов = 0, не трогает компоненты, только порядок в `DashboardRop`. Сохраняет привычку «всё на одной странице».
- **Cons:** layout всё ещё «cluttered» — 9 cards подряд без визуальных якорей. Krug: «РОП не поймёт где кончается «сегодня» и начинается «месяц»». Не решает Q5 целиком.
- **When:** когда времени = 2 часа и хочется минимально вмешаться.

### B: Three-section grouping «Сегодня / Качество / Тренды»

- **Суть:** добавить три заголовка секций (`<h2>` с ai-gradient underline), сгруппировать блоки по вопросам РОПа:
  1. **«Сегодня — кто работает»** = Block 1 (Activity) + Block 3 (Worst calls) + Block 5 (Unfulfilled commitments).
  2. **«Качество — что хромает»** = Block 2 (Quality score) + Block 4 (Missing phrases) + Block 4b (Department patterns).
  3. **«Тренды — куда клиенты движутся»** = Block 7 (Funnel) + Block 6 (Heatmap).
- **Pros:** даёт mental model (Krug ✅), три якоря для глаз = три сценария утра. Не убивает блоки. Подсвечивает что Block 7 это про **результат**, а не про KPI воронки. 24h-feasible.
- **Cons:** требует styling section headers, нужен ревью на mobile (collapsing? нет — раскрытые). Если РОПу нужны heatmap+funnel вместе вверху — он их найдёт ниже, но scroll нужен.
- **When:** когда нужен баланс «структура + минимум кода». **Это default для премьеры**.

### C: Above-the-fold hero card + накопительный стрим

- **Суть:** в самом верху — компактная hero-карточка (1 строка) с агрегатами отдела за период: «Сегодня: 142 набора · 38 разговоров · средний скрипт 64% · 3 МОПа в красной зоне». Под ней — те же три секции что в B.
- **Pros:** Nir Eyal trigger в чистом виде: первая фраза = «всё ли ОК?». Если 3 МОПа в красной зоне → клик ведёт на Block 2 anchor. Это soft anomaly (цифры + цвет, не popup), не алёрт.
- **Cons:** новый компонент = 4-6 часов, риск что данные не сходятся с Block 1 (consistency check), может выглядеть как BI-дашборд (анти-канон #37). Нужна точная формулировка чтоб не уйти в выручку.
- **When:** если успеваем к 4.05 и есть бюджет на отдельный hero-блок. Иначе откладываем в P2.

### D: Tabs «Главная / Качество / Тренды» (анти-канон)

- **Суть:** разрезать главную на 3 таба — РОП кликает что хочет.
- **Pros:** нет clutter совсем, экран всегда чист.
- **Cons:** Krug: «РОПу нужно видеть всё в одном глазу, табы = лишний клик». Канон #37 говорит «главный экран = 5+4+3 блока на одной странице». Если РОП должен помнить какой таб где — это уже фича для analytics-юзера, не для РОПа в 9:00 утра.
- **When:** **не подходит** для diva. Откатывается.

## Decision from Nir Eyal

**Выбор: Option B (три секции), с подмешиванием мини-hero из C если успеваем в P1.**

### Почему:
1. **Trigger** = первая секция «Сегодня — кто работает». РОП утром приходит → ищет «есть ли проблемные МОПы СЕГОДНЯ?». Block 1 (table) сразу даёт ответ: 0 наборов у Иванова → красный/амбер pipelineGap → клик в drill-down. Это полный hook loop за 5 секунд.
2. **Action** = Block 3 (worst calls) сразу под Block 1. РОП увидел красного МОПа → ниже видит конкретные звонки этого МОПа → клик → карточка → решение «поговорить с Иваном после обеда». Investment = пошёл говорить с МОПом.
3. **Variable reward** = Block 5 (unfulfilled commitments) в той же секции «Сегодня». Иногда там пусто (награды нет), иногда 5 follow-up'ов с конкретными цитатами — вот ради этого РОП открывает вкладку завтра.
4. **Krug**: каждая секция = один вопрос. РОП не думает «а где про звонки из вчера», у него три якоря в голове.
5. **Анкета**: «алёрты не нужны» — мы их и не делаем. Soft anomaly = амбер цвет в Block 1 pipelineGap (уже есть) + красный в Block 2 (уже есть). Никакого modal / banner / sticky alert.

### Recommended block order (1→9 с обоснованием)

```
СЕКЦИЯ I — «Сегодня — кто работает» (above the fold)
  1. Block 1 — Daily activity per МОП (full-width table)
       → answers: "кто сегодня сколько сделал"
       → frequency: каждое утро (anketa §9)
  2. Block 3 — Top-10 проблемных звонков (full-width list)
       → answers: "куда сейчас зайти и послушать"
       → frequency: каждое утро + по запросу
  3. Block 5 — Обещания требующие follow-up (full-width list)
       → answers: "кому позвонить из вчерашних"
       → frequency: каждое утро (variable reward — иногда пусто)

СЕКЦИЯ II — «Качество отдела» (mid-page)
  4. Block 2 — Оценка скрипта per МОП (half-width bar)  ─┐
                                                          ├─ grid 2-cols
  5. Block 4 — Упущенные техники (half-width list)      ─┘
       → answers: "куда системно хромает отдел"
       → frequency: 2-3 раза в неделю
  6. Block 4b — Системные паттерны (weakSpot ‖ criticalErrors, grid 2-cols)
       → answers: "что повторяется системно у нескольких МОПов"
       → frequency: раз в неделю

СЕКЦИЯ III — «Тренды и контекст» (below the fold)
  7. Block 7 — Куда движутся клиенты (full-width — РЕДИЗАЙН, см. ниже)
       → answers: "что в итоге происходит со сделками"
       → frequency: 2-3 раза в неделю
  8. Block 6 — Когда лучше звонить (heatmap, full-width)
       → answers: "когда поднимают трубку"
       → frequency: раз в неделю / при настройке расписания

ФУТЕР
  9. Last sync + pipeline gap status (как сейчас)
```

**Что изменилось vs текущий код:**
- Block 5 переехал из «середины» в первую секцию.
- Block 4b переехал из «после Block 3» в секцию «Качество» (где ему место по смыслу).
- Block 7 разлучён с Block 6 (бывшая grid-пара) — Block 7 редизайнится, Block 6 уходит в самый низ как самый редкий по частоте.

## Block 7 Redesign — «Куда движутся клиенты»

### Проблема текущего варианта
Сейчас Block 7 = вертикальный список с горизонтальными прогресс-барами (`linear-gradient(135deg, var(--ai-1), var(--ai-2))`). Все стадии равной толщины, gradient одинаковый, цифры справа. Это **chart-junk на лестнице** — нет ощущения движения, не видно «много vs мало», нет последовательности этапов воронки.

### Что чинит редизайн
Это не «конверсия %», это «распределение сделок по стадиям из числа тех, у кого был звонок». Нам нужно показать **массу** на каждом этапе (а не % перехода между этапами). Поэтому правильный визуал — **horizontal stacked bar** (одна полоса = весь period, сегменты = стадии).

### Wireframe — Option 1 (рекомендуемый): Horizontal stacked bar + leaderboard

```
┌──────────────────────────────────────────────────────────────────────┐
│ Куда движутся клиенты после наших звонков                            │
│ Сделки со звонками за период — где они сейчас сидят (без % конверсии)│
│                                                                      │
│ Всего: 142 сделки                                                    │
│                                                                      │
│ ┌──┬──────────┬─────────────────┬────────┬────┬──────┬──────┐       │
│ │НР│ Квалиф.  │ Презентация     │ Оплата │Win │ Lost │др.   │       │
│ │8 │   34     │       58        │   18   │ 14 │  6   │  4   │       │
│ └──┴──────────┴─────────────────┴────────┴────┴──────┴──────┘       │
│   6%   24%         41%             13%   10%   4%    3%             │
│                                                                      │
│ Top-3 этапа за период:                                               │
│  • Презентация ......... 58 (41%) ── основная масса висит здесь      │
│  • Квалификация ......... 34 (24%)                                   │
│  • Оплата ............... 18 (13%)                                   │
│                                                                      │
│ Финальные:                                                           │
│  ✅ Win  14 · ❌ Lost 6 · WL ratio 2.3                              │
└──────────────────────────────────────────────────────────────────────┘
```

**Реализация:**
- Один div с flex-row, сегменты пропорционально count.
- Первые 4 «бизнес-этапа» — gradient ai-1→ai-2 разной интенсивности (`bg-ai-1/30`, `bg-ai-1/60`, `bg-ai-2/60`, `bg-ai-2/90`).
- Win = `status-green/40`, Lost = `status-red/30`, «другие» (всё что > 6 этапов) сворачивается в один сегмент `surface-3`.
- Под полосой — leaderboard top-3 + Win/Lost счётчик.
- Подпись «без % конверсии» — это honest, чтоб РОП не путал с CRM-конверсией.

**Pros:**
- Krug: «один взгляд = одна история». Полоса = сразу видно где масса.
- Tufte: data-ink ratio высокий — каждый сегмент = one-shot count.
- Nir Eyal: trigger = «41% висит в Презентации?! почему не в Оплате» → клик в /managers с фильтром по этапу.
- Не врём про % конверсии (которых у нас нет, потому что нет stage history).

**Cons:**
- Маленькие этапы (3-5 сделок) превращаются в узкую полоску, которую плохо видно. Решение: «другие» сегмент-merge (см. выше).
- Названия этапов не помещаются в полосе при mobile → fallback на список под полосой.

### Wireframe — Option 2 (альтернатива): Mini-funnel «висят / финал»

```
┌──────────────────────────────────────────────────────────────────────┐
│ Куда движутся клиенты после наших звонков                            │
│                                                                      │
│ В работе (118):                                                      │
│   ░░░░░░░░░░░░░░░░░░░ Не разобрано     8                             │
│   ▓▓▓▓▓▓▓▓▓▓ Квалификация              34                            │
│   ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓ Презентация      58                            │
│   ▓▓▓▓▓ Оплата                         18                            │
│                                                                      │
│ Финал (24):                                                          │
│   ✅ Win  14    ❌ Lost  6    🔄 Возврат ушедших  4                 │
└──────────────────────────────────────────────────────────────────────┘
```

Это разрезает воронку на «висит / закрылось», что РОП и хочет видеть. Но менее «один-glance» чем Option 1.

### Anti-pattern (что НЕ делать)
- ❌ **Sankey** — у нас нет stage transition log, только текущая стадия. Sankey соврёт.
- ❌ **% между стадиями (CRM-style funnel)** — anti-canon #37 («конверсия стадий есть в CRM»).
- ❌ **Pie chart** — больше 5 сегментов = нечитаемо. Anti-pattern (Tufte).

**Рекомендация:** Option 1 (stacked bar). 4-6 часов работы.

## Today's Red МОП — anomaly callout?

**Вердикт: НЕ делать отдельный hero callout «сегодня плохо у Васи».**

### Почему
1. **Anketa §9 прямо говорит:** «Алерты не нужны». Hero callout = алёрт.
2. Block 1 уже показывает аномалию через цвет: pipelineGap → амбер/красный, low scriptScorePct → красный в Block 2. Этого достаточно для триггера, не нужен дублирующий callout.

## Risks (что учесть при имплементации)

1. **Section headers не должны выглядеть как navigation.** Это не nav, не tabs, не collapsible. Просто типографический маркер (h2 + ai-1 underline 80px). Иначе РОП будет искать клик и не найдёт.
2. **Block 7 stacked bar — fallback при ≤ 2 стадиях.** Если в `getDealStagesAfterCalls` вернулась 1 стадия — рисовать leaderboard без bar (полоса из одного сегмента бессмысленна).
3. **Block 5 в первой секции — если пусто, не показывать заглушку «нет обещаний».** Это пустое место = bad reward. Лучше скрывать карточку целиком когда `items.length === 0`. Сейчас код показывает «Нет открытых обещаний за период» — это гасит variable reward. Заменить на null.
4. **Soft summary строка** — пересчёт на server side, не дублировать `daily.length`. Считать red zone через `daily.filter(r => (r.scriptScorePctAvg ?? 1) < 0.5).length`.
5. **Не сломать period switcher.** PeriodFilter (today / week / month) сейчас дёргает все блоки. После reorder нужно прокликать каждый период на demo URL чтоб убедиться что данные грузятся.
6. **Mobile** — три секции = три h2. На < 1024px вертикальная высота вырастет на ~150px. Проверить scroll perf.
7. **Аnketa-проверка перед мерджем.** Перед commit прочитать §9 ещё раз: «алерты не нужны» — убедиться что summary-строка читается как summary, не как алёрт.
